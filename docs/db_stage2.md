# SQLite Stage 2

## Что добавлено

- Добавлен мягкий sync master-вкладки Google Sheets `Разбор потерь исх. потока` в SQLite.
- Источник кейсов для stage 2: Google Sheets, Yandex Disk на этом этапе не участвует в заполнении `cases`.
- Sync подключён на старте приложения после `init_db()` в безопасном `try/except`: падение sync не валит Telegram-бота.

## Как теперь работает sync

- Сервис: `bot/services/case_sync.py`
- Точка входа: `sync_cases_from_master_sheet(...)`
- При запуске sync:
  - открывает master-вкладку;
  - проверяет обязательные business headers;
  - проверяет наличие техколонки `case_id`;
  - если `case_id` отсутствует, добавляет колонку в конец header row;
  - если у строки пустой `case_id`, генерирует `uuid4().hex` и записывает только в нужную ячейку;
  - читает непустые строки;
  - нормализует row в канонический snapshot;
  - считает `row_hash` только по business snapshot, без `case_id` и служебных полей;
  - пишет актуальную карточку кейса в `cases`;
  - сохраняет историю изменений в `case_versions`;
  - сохраняет сырой снимок строки в `raw_sheet_rows`;
  - сохраняет item-данные строки в `case_items`;
  - обновляет `sheet_sync_state`;
  - пишет статус в `imports`.

## `case_id`

- `case_id` является стабильным техническим ключом кейса.
- Если в листе колонка уже есть, sync использует существующие значения.
- Если колонки нет, она добавляется последней, без перестановки business-колонок.
- Если у строки значение пустое, генерируется новый `case_id` и пишется обратно только в эту ячейку.
- Уже существующие `case_id` никогда не перезаписываются.

## Нормализация строки

- Заголовки приводятся к нормализованному виду:
  - нижний регистр;
  - сжатие лишних пробелов;
  - `ё -> е`;
  - поддержка алиасов вроде `Коментарий` / `Комментарий`.
- Пустые значения переводятся в `None`.
- Обязательные поля кейса:
  - `Дата разбора`
  - `Аналитик`
  - `Наименование`
  - `ID виновного`
  - `Коментарий` / `Комментарий`
  - `Пример попутного/обработанного шк`
  - `Что предпринято`
  - `Движение товара`
- Дополнительные поля (`ШК`, `Тара/передача`, `Сумма`, `Количество ШК`, `Склад`, `Отчет/Запрос`, даты движения и списания и т.д.) сохраняются в snapshot/raw и, где применимо, в `cases`/`case_items`.

## Защита от дублей

- `case_versions`
  - новая версия добавляется только если для `case_id` ещё нет такого `row_hash`;
  - есть уникальный индекс `(case_id, row_hash)`.
- `raw_sheet_rows`
  - дубль не добавляется, если уже есть запись с теми же `sheet_name`, `row_number`, `row_hash`;
  - есть уникальный индекс `(sheet_name, row_number, row_hash)`;
  - `is_latest` переключается без создания дубликата даже если строка вернулась к старому состоянию.
- `cases`
  - `upsert_case(...)` обновляет карточку только при реальном изменении полей/позиции строки;
  - повторный sync по неизменённой строке не делает бессмысленный update.
- `case_items`
  - повторный sync не плодит одинаковые item-записи;
  - дедуп выполняется по `case_id + shk + tare_transfer + item_name + last_movement_at`;
  - при изменении суммы/количества/сопутствующих полей существующий item обновляется.
- При миграции stage 1 -> stage 2 существующие дубликаты в `case_versions`, `raw_sheet_rows`, `case_items` предварительно чистятся перед созданием уникальных индексов.
